🧩 Unity Plugin

The core data layer for intergroup management. Unity defines the domain model — Groups, Meetings, Members, Positions, Locations, Contacts, and Intergroup Meetings — as a set of PHP interfaces, repositories, and factories, wired together through a dependency injection container.

What Unity Does

Unity is an abstract framework. It defines what a Group, Meeting, or Member looks like (via interfaces) and how data is accessed (via repositories), but it does not provide concrete implementations. A companion plugin — typically one that bridges a specific data source like TSML or ACF — hooks into Unity's unity/register_services action to provide the real implementations.

Every other plugin in the suite (Integrity, Reconcile, Scrutiny, Trumpet, Amber) depends on Unity as the shared data layer. They resolve services from Unity's container and work with the same interfaces, regardless of the underlying storage.

Requirements

  • WordPress 6.0 or later
  • PHP 8.0 or later
  • Advanced Custom Fields (ACF) — recommended for field configuration (JSON setup files included)
  • A service implementation plugin that hooks into unity/register_services and registers concrete factories and repositories

Installation

  1. Upload the unity folder to /wp-content/plugins/, or install via the WordPress plugin uploader.
  2. Activate Unity in the Plugins screen.
  3. Import ACF field groups from the setup/ directory (see ACF Field Groups).
  4. Install and activate a service implementation plugin that registers concrete services via the unity/register_services hook.
  5. Activate any dependent plugins (Integrity, Reconcile, Scrutiny, Trumpet, Amber) as needed.
⚠️
Unity alone won't do anything visible. Without a plugin that registers service implementations, resolving any service will throw a DependencyNotRegisteredException. This is by design — Unity is a framework, not a standalone application.

Boot Sequence

Unity initialises in a specific order during WordPress's plugins_loaded action (priority 10):

plugins_loaded
→
initContainer()
→
unity/register_services
→
initServices()
→
unity/loaded
StepWhat Happens
1. initContainer()Creates the DependencyContainer singleton. The UnityServiceProvider registers default (placeholder) factories for every service — each one throws DependencyNotRegisteredException until overridden.
2. unity/register_servicesAction hook. The implementation plugin listens here and re-registers each service with a real factory callable. The container is passed as the hook parameter.
3. initServices()Resolves all core services from the container, triggering lazy instantiation. If any service hasn't been registered by step 2, the placeholder throws an exception and an admin notice is shown.
4. unity/loadedAction hook. All services are now available. Other plugins (Integrity, Scrutiny, etc.) listen here to initialise themselves. The container is passed as the hook parameter.

Dependency Container

Unity includes a lightweight dependency injection container (DependencyContainer) that implements the Container interface. It supports three operations:

MethodDescription
register(string $id, callable $factory)Register a factory callable. Called with the container as its argument, allowing services to resolve their own dependencies.
get(string $id): mixedResolve a service. The factory is called only once — subsequent calls return the cached singleton. Throws RuntimeException if the service is not registered.
has(string $id): boolCheck if a service or factory is registered.

Services are identified by their interface class name (e.g. GroupRepository::class). This means consumer code depends on interfaces, not concrete classes.

// Register a service (in your implementation plugin)
$container->register(GroupRepository::class, function (DependencyContainer $c) {
    return new MyGroupRepository(
        $c->get(GroupFactory::class),
        $c->get(Cache::class)
    );
});

// Resolve a service (anywhere after unity/loaded)
$groupRepo = unity()->get(GroupRepository::class);

Service Provider

The UnityServiceProvider registers placeholder factories for every service during initContainer(). Each placeholder throws DependencyNotRegisteredException to give a clear error message if the implementation plugin fails to register a service.

Only two services have real default implementations built into Unity:

ServiceDefault Implementation
CacheWordPressCache — wraps wp_cache_get, wp_cache_set, wp_cache_delete, wp_cache_flush.
ConfigurationUnityConfiguration — simple in-memory key/value store for cross-plugin configuration.

All other services (every Factory, Repository, ChangeTracker, and ViewFactory) must be registered by the implementation plugin.

WordPress Hooks

unity/register_services

Fires after the container is created but before services are resolved. This is where your implementation plugin registers concrete factories and repositories.

Parameter: DependencyContainer $container

Priority: default (10). If you need to override another plugin's registration, use a higher priority.

unity/loaded

Fires after all services are resolved and Unity is fully operational. Other plugins in the suite (Integrity, Reconcile, Scrutiny, Trumpet, Amber) listen on this hook to initialise themselves.

Parameter: DependencyContainer $container

💡
If nobody hooks into unity/register_services, Unity detects this and shows an admin notice: "Services not registered". This typically means the implementation plugin is missing or deactivated.

Design Patterns

Interface-Driven Design

Every entity (Group, Meeting, Member, etc.) is defined as a PHP interface in the Interfaces subdirectory of its module. Consumer code depends only on interfaces — never on concrete classes. This allows completely different storage backends (ACF, custom tables, external APIs) to implement the same contracts.

Repository Pattern

Data access goes through Repository interfaces. Each repository provides find, findAll, save, delete, and domain-specific query methods (e.g. findByDay, findOnline). Repositories are resolved from the container and are typically backed by WP_Query and post meta.

Factory Pattern

Object creation goes through Factory interfaces. Factories typically accept a WordPress post ID and hydrate a domain object from post meta and ACF fields. Some factories also provide a createNew method for building objects from raw values (used by the Reconcile importer).

Dependency Injection

Unity's container wires everything together. Factories receive the container in their registration callable and use it to resolve their own dependencies (e.g. a MeetingFactory needs a ContactFactory and a LocationRepository). Services are singletons — created once, reused everywhere.

Change Tracking

Unity defines ChangeTracker interfaces for Groups, Members, and Positions. These track edits made through ACF and fire hooks when real changes are detected, enabling plugins like Trumpet to send notifications.

Each tracker follows the same two-step pattern:

  1. captureOriginalGroup/Member/Position(int $postId) — snapshot the entity before ACF saves the fields
  2. checkForChanges(int $postId) — compare the saved entity to the snapshot and fire a hook if anything changed
💡
Change trackers are registered by the implementation plugin, not Unity itself. They typically hook into ACF's acf/save_post action to capture before/after state.

Accessing Services

Unity provides a global unity() helper function that returns the dependency container. Use it anywhere after the unity/loaded hook has fired:

// Get the container
$container = unity();

// Resolve services
$groupRepo    = $container->get(GroupRepository::class);
$memberRepo   = $container->get(MemberRepository::class);
$meetingRepo  = $container->get(MeetingRepository::class);
$positionRepo = $container->get(PositionRepository::class);

You can also access the container from the unity/loaded hook parameter:

add_action('unity/loaded', function (DependencyContainer $container) {
    $groups = $container->get(GroupRepository::class)->findAll();
});

$groupRepo = unity()->get(GroupRepository::class);

Creating Entities

Factories create domain objects from raw data. The most common use case is hydrating from a WordPress post ID:

$groupFactory = unity()->get(GroupFactory::class);
$group = $groupFactory->createFromSource(123); // from WP post ID

The MemberFactory also supports creating from raw values (used by the Reconcile importer):

$memberFactory = unity()->get(MemberFactory::class);
$member = $memberFactory->createNew(
    id: $postId,
    anonymousName: 'John D.',
    homeGroup: 15,
    isGSR: true,
    personalEmail: 'john@example.com'
);

Caching

Unity's Cache interface wraps WordPress's object cache (wp_cache_* functions). It supports keyed storage with optional groups and expiration:

$cache = unity()->get(Cache::class);

$cache->set('my_key', $data, 'my_group', 3600); // expires in 1 hour
$value = $cache->get('my_key', 'my_group');
$cache->delete('my_key', 'my_group');
$cache->flush(); // clear everything

Repository implementations typically use the cache to avoid redundant WP_Query calls. The default WordPressCache adapter delegates to WordPress's built-in cache system, so it benefits from persistent cache backends (Redis, Memcached) if configured.

Configuration

The Configuration interface provides a simple in-memory key/value store for sharing settings across plugins:

$config = unity()->get(Configuration::class);

$config->setConfig('meeting_types', ['Open', 'Closed', 'Speaker']);
$types = $config->getConfig('meeting_types'); // ['Open', 'Closed', 'Speaker']

Values are stored in memory for the current request only. The Configuration service is available to all plugins in the suite and is useful for sharing lookup tables, field mappings, or feature flags without touching the database.

Registering Services

Your implementation plugin must hook into unity/register_services and re-register every factory and repository with a real callable. Here's the typical pattern:

add_action('unity/register_services', function (DependencyContainer $container) {

    // Contact Factory
    $container->register(ContactFactory::class, fn() =>
        new MyContactFactory()
    );

    // Location Factory + Repository
    $container->register(LocationFactory::class, fn() =>
        new MyLocationFactory()
    );
    $container->register(LocationRepository::class, fn($c) =>
        new MyLocationRepository($c->get(LocationFactory::class))
    );

    // Meeting Factory + Repository
    $container->register(MeetingFactory::class, fn($c) =>
        new MyMeetingFactory(
            $c->get(ContactFactory::class),
            $c->get(LocationRepository::class)
        )
    );
    $container->register(MeetingRepository::class, fn($c) =>
        new MyMeetingRepository(
            $c->get(MeetingFactory::class),
            $c->get(Cache::class)
        )
    );

    // ... Group, Member, Position, IntergroupMeeting,
    //     ChangeTrackers, ViewFactories

});
🚨
Every factory and repository must be registered. If you skip one, any code that resolves it — including Unity's own initServices() — will throw a DependencyNotRegisteredException and the plugin suite won't load.

Error Handling

DependencyNotRegisteredException

Thrown when a service is resolved but no implementation plugin has registered a real factory. The exception message includes the missing class name. This is the most common error and means the implementation plugin is either missing, deactivated, or didn't register that particular service.

RuntimeException — "Service not found"

Thrown by the container when you try to resolve a service that was never registered at all — not even with a placeholder. This shouldn't happen with the standard service provider.

RuntimeException — "Plugin not initialized"

Thrown when Plugin::getContainer() or unity() is called before the boot sequence has run. Make sure you only access the container after the unity/loaded hook.

All exceptions during initialisation are caught, logged to the WordPress error log with full stack traces, and displayed as admin notices. Enable WP_DEBUG and WP_DEBUG_LOG to see detailed information.

Troubleshooting

"Services not registered" admin notice

No plugin is listening on the unity/register_services hook. The implementation plugin (the one that provides concrete factories and repositories) needs to be installed and activated.

"Dependency not registered: ClassName"

The implementation plugin didn't register a concrete factory for that service. Check that all required services are covered. See the Service Catalogue for the full list.

"Plugin not initialized" when calling unity()

You're accessing the container too early. Move your code to run on or after the unity/loaded hook, or use plugins_loaded with a priority greater than 10.

ACF fields not appearing

Import the ACF field groups from setup/Unity_ACF.json or setup/unity-prod-acf.json. Make sure the Advanced Custom Fields plugin is installed and activated.

Dependent plugins (Integrity, Reconcile, etc.) fail to load

These plugins listen on unity/loaded. If Unity fails during initialisation, this hook never fires. Check the WordPress error log for Unity-specific errors first.

FAQ

Why doesn't Unity include concrete implementations?

By design. Unity defines the contracts; a separate plugin provides the implementation. This means you can swap the data source (ACF, custom tables, external API) without changing any of the dependent plugins. Every plugin in the suite works with the same interfaces.

Can I override a single service?

Yes. Hook into unity/register_services at a higher priority than the implementation plugin and re-register just the service you want to replace. The container uses the last-registered factory.

Are services singletons?

Yes. The container calls the factory callable only once. After that, the same instance is returned for every get() call.

What's the unity() helper function?

A global shorthand for Plugin::getContainer(). It returns the DependencyContainer singleton so you can resolve services from anywhere.

What are the 7th Tradition contribution fields on Groups?

Groups can store Venmo, PayPal, and Square handles for digital contributions (a common practice in AA intergroups). The hasContributionOptions() method returns true if any of these are set.

Do I need ACF?

ACF is recommended but not strictly required. The provided JSON setup files configure ACF field groups for all entity types. If you're building a custom implementation that doesn't use ACF, you can store data however you like — Unity only cares that your factories and repositories implement the correct interfaces.
